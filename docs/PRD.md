# Product Requirements Document

## 1. Executive summary

AI Intelligence OS is a local-first research intelligence workspace that turns a high-volume stream of AI papers and software releases into a small number of trustworthy, technically deep, buildable, and commercially relevant intelligence products.

The product solves four connected problems:

1. **Discovery overload:** new AI material appears faster than one person can monitor it.
2. **Shallow understanding:** summaries rarely explain method, evidence, implementation, and limitations deeply enough to build from.
3. **Fragmentation:** papers, code, model cards, reviews, releases, and commercial implications live separately.
4. **Knowledge decay:** useful understanding is lost across tabs, chats, and unstructured notes.

## 2. Product principles

- Evidence before eloquence.
- Deterministic processing before LLM processing.
- Depth for a few items instead of shallow coverage of everything.
- Local-first privacy and zero recurring service cost.
- Human-verifiable output; no false promise of perfect accuracy.
- Cumulative intelligence: today's work must improve tomorrow's retrieval and ranking.
- Resource-aware execution on the specified laptop.

## 3. Personas

### P1 — Builder/learner

Needs to understand techniques deeply enough to implement or evaluate them. Values diagrams, equations, code links, reproduction requirements, caveats, and prerequisites.

### P2 — Product/consulting founder

Needs to translate technical change into a painful workflow, defined buyer, demonstrable product, paid pilot, and credible advantage.

Both personas are currently the same user, but their scores and report sections remain separate.

## 4. Jobs to be done

- When important AI work appears, tell me what materially changed and why.
- When a paper makes a claim, show the evidence and conditions behind it.
- When code exists, show whether it appears maintained and usable.
- When I want to build it, give me an implementation and reproduction path suited to my hardware.
- When a technique has business potential, map it to a buyer and workflow without inventing market evidence.
- When I return later, let me retrieve the accumulated understanding and see what changed.

## 5. Functional requirements

### FR-01 Source registry

The system shall maintain versioned configuration for each source: key, trust tier, base URL, enabled state, polling interval, rate limit, cursor, terms link, parser version, and health state.

Acceptance:

- Source configuration is data-driven, not a hardcoded conditional chain.
- A source can be disabled without code changes.
- Credentials, if ever needed, come from environment variables and are never stored in content or logs.

### FR-02 Incremental ingestion

The system shall fetch new or changed metadata using a saved cursor or time window and an overlap period to prevent missed records.

Acceptance:

- Repeating the same fetch creates no duplicate work.
- Partial failures preserve successfully normalized records and a resumable checkpoint.
- HTTP 429/5xx/network errors use bounded exponential backoff with jitter.
- Raw response fixtures can replay ingestion offline.

### FR-03 Canonical identity and versioning

The system shall identify works across source-specific IDs and retain revisions.

Identity precedence for papers:

1. DOI when valid.
2. Normalized arXiv identifier without version.
3. OpenReview forum/submission ID.
4. Conservative fingerprint from normalized title, first author, and year.

Acceptance:

- Revision `v2` links to the same work as `v1` but creates a new version record.
- A possible fuzzy duplicate is queued for review instead of being merged destructively.
- Canonical URLs are generated by connector code, not the LLM.

### FR-04 Document acquisition and parsing

The system shall download full text only for shortlisted candidates or explicit user requests.

Acceptance:

- File type, maximum size, hash, source URL, and retrieval time are recorded.
- PDF text retains page and section provenance.
- Parsing failure does not discard metadata.
- Temporary low-value files are eligible for cleanup.

### FR-05 Topic classification

The system shall assign zero or more controlled topics plus an `unknown` state.

Acceptance:

- Deterministic source/category rules run first.
- Local model classification is schema-constrained.
- Classification records include model, prompt version, confidence, and explanation.
- Low-confidence results do not silently become high-relevance candidates.

### FR-06 Candidate ranking

The system shall calculate transparent technical and commercial rankings from normalized feature values.

Acceptance:

- Formula and component scores are stored.
- No LLM returns the final numeric score.
- Missing values use documented neutral/default behavior.
- Ranking weights are versioned and replayable.

### FR-07 Fast brief generation

The system shall create a structured short brief for leading candidates.

Required fields: one-sentence change, problem, claimed contribution, evidence available, limitations noticed, code availability, relevance, recommended action, sources.

Acceptance:

- Output validates against its JSON schema.
- Claims without evidence are marked unverified.
- Failed validation is repaired at most once, then stored as failed for review.

### FR-08 Deep technical analysis

The system shall generate deep dives from selected evidence chunks rather than placing an entire uncontrolled document in one prompt.

Required stages:

1. Section inventory.
2. Evidence extraction.
3. Method analysis.
4. Evaluation/benchmark analysis.
5. Code and reproducibility enrichment.
6. Production implications.
7. Commercial hypothesis.
8. Skeptical review.
9. Final synthesis.

Acceptance:

- Intermediate structured artifacts are persisted.
- Every major factual claim points to one or more evidence IDs.
- Reviewer can reject, qualify, or request regeneration of a section.
- Final synthesis cannot add new factual claims absent from intermediate evidence.

### FR-09 Code/repository enrichment

For shortlisted works, the system shall connect official or author-linked repositories and capture objective signals.

Signals: canonical repository URL, claimed relationship, license, default branch, latest commit time, releases, archived state, dependency manifests, installation instructions, tests, open issues count, and explicit hardware statements.

Acceptance:

- Popularity is not treated as correctness.
- No repository code is executed in phase one.
- Repository relationship is labelled official, author-linked, community, or unknown.

### FR-10 Skeptical review

The system shall review reports independently for unsupported claims, unfair comparisons, missing baselines, data leakage concerns, absent ablations, reproducibility gaps, source conflicts, and overconfident business conclusions.

Acceptance:

- Reviewer output is structured and cannot silently edit evidence.
- Critical findings prevent `verified` status.
- Rejected or qualified claims remain auditable.

### FR-11 Knowledge retrieval

The user shall search by keyword, semantic similarity, topic, author, organization, date, source, report status, implementation availability, and commercial score.

Acceptance:

- Keyword search works without an embedding model.
- Search results expose why each result matched.
- Retrieval respects deleted/invalidated versions.

### FR-12 Daily workspace

The Today screen shall show:

- Date and pipeline status.
- Number discovered, filtered, shortlisted, analyzed, and failed.
- Top technical developments.
- Top commercial opportunities.
- Deep-dive cards.
- Important revisions/releases.
- Connector or resource warnings.

### FR-13 Deep-dive reader

The reader shall provide:

- Sticky table of contents.
- Confidence and review status.
- Technical sections.
- Evidence drawer with source/page/section.
- Code/reproduction tab.
- Business opportunity tab.
- Prerequisite learning tab.
- User feedback: useful, irrelevant, too shallow, incorrect, follow topic, mute topic.

### FR-14 Personalization

The system shall update explicit topic preferences and record feedback events. Phase one personalization adjusts ranking weights conservatively; it does not train a model.

### FR-15 Scheduling and manual control

The user shall run discovery, ranking, deep analysis, and report generation independently from the UI/CLI. Scheduled runs must skip expensive stages when the laptop is on battery or system resource thresholds are exceeded, when detectable.

### FR-16 Resource governor

The system shall enforce one LLM task at a time, queue depth limits, download concurrency, document size limits, context budgets, and storage cleanup policies.

### FR-17 Auditability

Every pipeline run shall record configuration version, connector version, ranking version, prompt version, model identity, timestamps, status, retry count, input references, output references, and errors.

## 6. Non-functional requirements

### Reliability

- All ingestion writes are idempotent.
- Processed state changes occur within transactions.
- Jobs are restartable from persisted stages.
- Corrupted documents are isolated rather than crashing the batch.

### Performance

- Metadata operations must not load model weights.
- Lists use pagination and indexed queries.
- Embeddings run in bounded batches.
- Long reports stream from the API or load section-wise.

### Security

- External content is untrusted.
- HTML is sanitized before UI rendering.
- No `shell=True`, dynamic `eval`, unsafe deserialization, or arbitrary file paths.
- Downloads remain inside a configured data root.
- URLs allow only HTTP(S) and pass connector/domain policy.
- Logs redact tokens and query secrets.

### Maintainability

- Domain services depend on protocols/interfaces rather than concrete connectors or model runtime.
- Source-specific transformations stay inside connectors.
- JSON schemas are versioned.
- Database migrations are explicit and reversible where feasible.

### Accessibility and UI

- Keyboard navigation and visible focus states.
- Semantic headings and landmarks.
- Text contrast meeting WCAG AA targets.
- Reduced-motion support.
- Desktop layout optimized for 1366px and above; useful behavior down to 360px.

## 7. System modules

| Module | Responsibility |
|---|---|
| `sources` | registry, connector protocol, source configuration |
| `ingestion` | polling, raw capture, checkpointing, retries |
| `catalog` | identity, normalization, versions, authors, topics |
| `documents` | downloads, hashing, parsing, evidence spans |
| `ranking` | feature extraction, weight versions, ranked lists |
| `inference` | Ollama adapter, model profiles, JSON validation, queue |
| `analysis` | fast brief, deep dive stages, reviewer workflow |
| `repositories` | GitHub enrichment and paper-code relationships |
| `search` | FTS, optional vectors, filters, result explanations |
| `reporting` | daily report assembly and export |
| `scheduler` | batch plans and resource-aware execution |
| `observability` | structured logs, job events, resource snapshots |
| `api` | FastAPI endpoints and error contracts |
| `web` | React workspace |

- `GET /health`
- `GET /system/resources`
- `GET /sources`
- `POST /sources/{key}/sync`
- `GET /runs`
- `GET /runs/{id}`
- `POST /pipeline/discover`
- `POST /pipeline/rank`
- `POST /items/{id}/brief`
- `POST /items/{id}/deep-dive`
- `GET /items`
- `GET /items/{id}`
- `GET /items/{id}/evidence`
- `GET /reports/daily`
- `GET /reports/{id}`
- `POST /feedback`
- `GET /search?q=...`

All mutations return a persisted job/run identifier instead of holding an HTTP request open for lengthy analysis.

## 9. State machines

### Work state

`discovered -> normalized -> shortlisted -> acquired -> parsed -> briefed -> analyzed -> reviewed -> verified`

Terminal/side states: `filtered`, `failed`, `rejected`, `superseded`.

Transitions are enforced in domain code and recorded as events.

### Job state

`queued -> running -> succeeded | failed | cancelled | deferred`.

## 10. Phase-one release acceptance

The release is acceptable when a clean installation can:

1. Sync arXiv and OpenReview fixtures and then live metadata.
2. Re-run without duplicating records.
3. Rank candidates with inspectable component scores.
4. Download and parse a selected paper with page-level evidence.
5. Use a local Ollama model to generate schema-valid briefs.
6. Produce at least one deep dive with skeptical findings and citations.
7. Enrich a linked GitHub repository without executing it.
8. Present Today, Explore, Deep Dive, and System screens in a polished responsive UI.
9. Run on the target machine within declared budgets.
10. Complete all golden-set, connector, API, and UI smoke tests.
